# ROADMAP — Клиент (EFIR Player)

Куда движется плеер и как будут реализованы будущие фичи. Серверная половина — `ROADMAP.md` в репозитории Server. Текущие хвосты до production — `TODO_Client.md`.

## Ключевое архитектурное решение: контент-модель v2 («сцены и зоны»)

Сейчас плеер умеет одно: полноэкранный слот = медиа или плейлист. Все будущие фичи (виджеты, музыка, стримы, idle-экран) упираются в это ограничение, поэтому сначала — платформа:

**Манифест v2** (обратносовместимо, новые типы рядом со старыми):
```jsonc
{
  "content_type": "layout",            // media | playlist | layout | stream | widget
  "layout": {
    "preset": "video+ticker",          // пресет или custom
    "zones": [
      { "rect": [0,0,100,92], "z": 0, "content": {"type":"playlist","id":3} },
      { "rect": [0,92,100,8], "z": 1, "content": {"type":"widget","kind":"ticker","config":{...}} }
    ]
  },
  "audio": { "playlist_id": 7, "volume": 60, "policy": "duck" },   // фоновая музыка
  "control": { "maintenance": false, "idle_policy": "logo", "service_mode_locked": false }
}
```

**Рефактор плеера — `ContentRenderer`.** `player_screen.dart` (1900 строк) разбирается на рендереры с единым интерфейсом: `ImageRenderer`, `VideoRenderer` (текущая двух-плеерная gapless-логика), `PlaylistRenderer`, `WidgetRenderer`, `StreamRenderer`, `LayoutRenderer` (Stack зон, каждая зона — снова renderer). Оффлайн-режим и онлайн перестают быть «двумя мирами»: оба собирают одинаковое дерево рендереров.

Почему так: это модель всех зрелых signage-решений — Xibo (layouts/regions), Yodeck (zones), OptiSigns (split screens). Мы берём путь Yodeck: **пресеты сцен вместо свободного конструктора** — «Полный экран», «Видео + бегущая строка», «Видео + часы в углу», «Сплит 2 зоны». Пресеты закрывают 90% случаев и не требуют от оператора думать.

## Фазы

### Phase 0 — Стабилизация (текущая, до боевого запуска)
Хвосты P0 из TODO: тесты ядра (manifest/cache/registration/sequencing), TTL-очистка кэша + контроль свободного места, экспорт диагностического пакета, коды ошибок, прогон на целевом железе (Smart TV), self-signed сертификаты.

### Phase 1 — Платформа контента v2
- Рефактор `player_screen` → ContentRenderer'ы (без новых фич, поведение 1:1).
- Парсинг манифеста v2 (layout/zones/control), фолбэк на v1.
- **Idle-экран** (политика `idle_policy` из манифеста): `black` (default — для витрин лучше чёрный), `logo` (asset организации из панели, кэшируется как media), `clock` (нативные часы). Сервисная информация (device id, QR) на публичный экран не выводится никогда — только в service-mode. Так делает Anthias (заставка только до настройки).
- Канал **удалённых команд**: ответ heartbeat `{commands:[…]}` + ACK; команды reload/clear_cache/restart/maintenance.

### Phase 2 — Виджеты
Нативные Flutter-виджеты (НЕ WebView: на Android 6 Smart TV встроенный браузер старый и тяжёлый — главное отличие от Xibo, у которого HTML-виджеты лагают на слабом железе):
- **Часы** — формат 12/24, дата, стиль (config из панели).
- **Бегущая строка** — свой текст из панели или RSS-лента (сервер проксирует и кэширует RSS, клиент получает готовые строки); скорость/цвет/фон.
- **Погода** — сервер-прокси к Open-Meteo (бесплатно, без API-ключа) по координатам экрана, кэш 15 мин на сервере; клиент рендерит нативно, оффлайн — последнее значение.
Виджеты живут в overlay-зонах поверх видео (пресеты сцен из Phase 1). Конфиг приходит в манифесте — обновление на лету без перезапуска.

### Phase 3 — Презентации и фоновая музыка
- **Презентации (pptx/pdf)**: конвертация на сервере (LibreOffice headless → PDF → PNG-страницы; embedded-видео извлекаются из pptx-архива). Клиент получает **обычный плейлист** изображений/видео с длительностями — нулевые изменения в плеере, работает на любом железе. (Так делают Yodeck/OptiSigns; рендер PDF на клиенте отклонён: шрифты и CPU старых TV.)
- **Фоновая музыка**: третий экземпляр `media_kit` Player (audio-only) поверх существующих двух видео-плееров. Политики: `duck` (приглушать музыку, когда видео со звуком), `mute_video`, `mix`. Аудио-файлы качаются тем же media-кэшем. Расписание музыки — обычные слоты с `audio`-блоком.

### Phase 4 — Прямые трансляции
- Клиент: `StreamRenderer` поверх mpv — HLS/DASH/RTSP играются по прямому URL «из коробки».
- YouTube/VK/RUTUBE — это страницы, а не потоки: **сервер резолвит** прямой HLS через yt-dlp (поддерживает все три платформы) и отдаёт клиенту URL с TTL. Протух/оборвался → клиент просит re-resolve → при недоступности показывает fallback-слайд «трансляция скоро начнётся» и ретраит по backoff.
- UX: оператор вставляет обычную ссылку на трансляцию в слот — всё остальное автоматически; панель показывает статус «поток доступен».

### Phase 5 — Управление питанием
- **MVP без железа** (закрывает ~70%): расписание «экран вкл/выкл» в панели → команда по heartbeat → клиент гасит дисплей: Android TV — HDMI-CEC standby (телевизор реально выключается), Windows — `SC_MONITORPOWER`, Linux — DPMS off. Плеер продолжает работать «за тёмным экраном».
- **Фаза с железом**: умная розетка/реле на ESP (Tasmota/ESPHome) + MQTT-брокер (Mosquitto в compose сервера); топики `efir/power/{screen_id}`; сервер шлёт on/off по расписанию; Wake-on-LAN для PC-плееров. Коммерческие панели (LG/Samsung) — RS232/IP-протоколы, как у Xibo/MagicInfo — по запросу.

### Phase 6 — Эксплуатация
Self-update (Windows installer `/VERYSILENT`, Linux-пакет, Android prompt — по расписанию вне эфирного времени, как у Anthias), `sentry_flutter` → self-hosted Sentry/GlitchTip, secure storage токена, расширенная телеметрия (uptime, clock_drift, disk_free).

### Phase W — Web-плеер для Smart TV (LG webOS / Samsung Tizen)
Телевизоры LG/Samsung получают плеер **без отдельного приложения**: лёгкое веб-приложение (ванильный JS/Preact + HTML5 `<video>`), которое раздаёт сам сервер по `/player`; на TV ставится только обёртка-ярлык (`.ipk` для webOS / `.wgt` для Tizen, собираются в CI). Тот же device API и тот же flow регистрации — в панели такие экраны неотличимы от Flutter-клиентов (`client_version: "web-…"`).

Flutter Web сознательно не используется: TV-браузеры старые (Chromium 53–79), CanvasKit тяжёлый, media_kit на web не работает. Web-плеер — **второй раннер той же контент-модели**: JS-рендереры повторяют ContentRenderer'ы Phase 1 (image/video/playlist/widget/layout). Ограничение по-честному: квоты браузерного кэша на TV малы → streaming-first, полноценный оффлайн остаётся преимуществом Flutter-клиента; старые Android TV продолжают жить на нём. Серверная половина — ROADMAP сервера, Phase W.

## Архитектурные долги клиента (что чинит Phase 1)
- `player_screen.dart` 1.9k / `editor_screen.dart` 1.5k / `playlist_controller.dart` 1.4k строк — монолиты без слоёв.
- Онлайн (`ManifestItem`) и оффлайн (`PlaylistItem`) — два параллельных мира с дублированной логикой воспроизведения.
- Тестов нет (placeholder) — рефакторить без них рискованно, поэтому Phase 0 раньше Phase 1.
- Токен в plaintext (`device.json`, chmod 600) → secure storage в Phase 6.
- Манифест-полл без ETag (серверная задача, клиент уже умеет «unchanged»).
